2 How sending + tracking works now

Sending — outbound

Tracking — inbound (replaces pixel + ngrok)

👁
EngineMailer
detects open/click
→
🔗
Webhook POST
JSON → /webhooks/engine-mailer (signed)
→
🗄️
DealFlow
match by activityId → bump openCount / clickCount

Each outgoing email carries a DealFlow activityId in a custom header/tag; the webhook echoes it back so we update the right activity. Same openCount/clickCount/firstOpenedAt fields — just fed by webhook instead of a pixel. The Engagement timeline on the activity page is unchanged.

3 What changes in the codebase

🗑 Removed

  • SMTP section in Settings (Gmail/Outlook/Yahoo)
  • SmtpEmailProvider + nodemailer wiring
  • /track/open/:token & /track/click/:token
  • email-tracking-token.ts (HMAC pixel tokens)
  • pixel/redirect injection in email-html-wrap.ts
  • the ngrok dependency for tracking

✦ Added

  • EngineMailerEmailProvider (REST via fetch)
  • EngineMailer block in OrgIntegrationsRepo (encrypted API key)
  • EngineMailerIntegrationSection in Settings
  • POST /api/v1/webhooks/engine-mailer (signed ingest)
  • webhook → activity mapper (open/click/bounce)
  • one-time migration prompt for orgs on old SMTP

The EmailProvider interface and the per-org encrypted integrations store already exist — so this is mostly a new adapter + a webhook route, not new architecture.

Spam / Complaintflag the activity; stop further sends to that address (future)

Accuracy caveat (unchanged by provider): opens are still pixel-based, so Apple Mail Privacy Protection & image-blocking inflate/miss opens. Clicks are the trustworthy signal — the UI will lean on clicks.
