DealFlow — Email Attachments v1

Architecture, storage layout, UI mockups, and recipient-side view for the proposed attachment feature.

1. End-to-end data flow

Browser (compose dialog)
→
user action Clicks + Attach files, picks proposal.pdf + diagram.png. Files stay in browser memory.
Browser (compose dialog)
→
user action Clicks Send email. Browser builds a multipart/form-data POST with one JSON field body + N binary file fields.
API · POST /api/v1/emails
→
parse @fastify/multipart streams each file to memory or disk. Validates size (≤ 25 MB each, ≤ 25 MB total) and MIME (blocks .exe, .bat, .vbs, …).
DB · activities + email_attachments
→
insert Pre-creates the email activity (subject, body, cc/bcc, tracking, …). Inserts one email_attachments row per file with the chosen storage path.
Disk · apps/api/data/uploads
→
write Writes each file to uploads/<orgId>/<activityId>/<uuid>-<sanitized-name>. Org-scoped subdir prevents cross-tenant leakage.
SMTP · nodemailer
→
send Sends the email with attachments: [{ filename, path }] — nodemailer reads from disk and attaches as multipart/mixed. Tracking pixel + click links still injected per v1 tracking flow.
Recipient inbox
→
delivered Recipient sees the email body plus a download list of all attachments at the bottom (standard MIME behavior).

If any file fails validation, the entire send is rejected with a 400 before anything writes to disk or DB. On send failure (SMTP error), files are deleted from disk and the activity row is marked delivery_status: 'failed'.

apps/api/data/
└── uploads/
    ├── f4b3c2…  # org id
    │   ├── a01923…  # activity id
    │   │   ├── 11ee29…-proposal-q3.pdf
    │   │   └── 22ff30…-architecture-diagram.png
    │   └── b02834…  # another email activity
    │       └── 33aa41…-pricing.xlsx
    └── e8d1a7…  # other org's files (isolated)

• Org-scoped subdirs mean a bug in path construction can leak at most within one tenant.
• UUID prefix prevents collisions when two emails have files with the same name.
• Filename gets sanitized (strip ../, control chars, etc.) before write.
• apps/api/data/ is added to .gitignore. Operators back up data/ alongside the Postgres dump.

3. DB schema — new table

CREATE TABLE email_attachments (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  activity_id uuid NOT NULL REFERENCES activities(id) ON DELETE CASCADE,
  filename text NOT NULL,  -- original filename (sanitized for display only)
  mime_type text NOT NULL,
  size_bytes integer NOT NULL,
  storage_path text NOT NULL,  -- relative to data/uploads/, e.g. "<orgId>/<activityId>/<uuid>-name.pdf"
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX email_attachments_activity_idx ON email_attachments (activity_id);

5. Validation errors (shown inline in the dialog)

File too large. video.mp4 is 32 MB; the limit is 25 MB per file. Skipped.
Blocked file type. installer.exe can't be attached for security reasons (executable file).
Total size limit reached. Adding screenshots.zip (8 MB) would exceed the 25 MB total. Remove a file first.

All client-side checks. The server re-validates and rejects with HTTP 400 if anything slips through (e.g., user disables JS).

8. Cleanup & lifecycle

Activity deleted → DB cascade deletes email_attachments rows. A small background hook unlinks the corresponding files from disk. If the file is already gone, the unlink is a no-op (idempotent).
Org deleted → Cascade deletes everything below it. Files orphaned on disk are caught by a once-a-day cleanup pass that walks data/uploads/<orgId>/ against the DB.
Send failure → Activity row marked delivery_status: 'failed', attachment rows + files retained so the user can retry by composing a new email (copy-attachment endpoint deferred to v2).
Disk backup → data/uploads/ backed up alongside Postgres dump. Both must be restored together to keep DB rows and files in sync.

9. Explicitly out of scope (v1)

  • Inline images via cid: references — images attach as files only.
  • Inbound attachments — depends on the deferred reply-detection sub-plan.
  • Virus scanning — file content is trusted to be what the user uploads.
  • Pre-upload / draft attachments — files travel with the send request, not separately.
  • Re-attaching saved files — each compose starts fresh; no "Attach from prior email" affordance.
  • Resumable uploads — single multipart POST. Under 25 MB so resumability isn't worth the complexity.
  • Thumbnails / image previews — files show by filename + MIME icon, not rendered preview.
  • S3 / object-storage backend — local filesystem only. A future sub-plan can add S3 as a pluggable backend.
