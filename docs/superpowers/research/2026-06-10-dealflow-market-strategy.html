DealFlow · product strategy

DealFlow is the AI-native, WhatsApp-first CRM that fills itself in — built for Malaysian and SEA SMB sales teams of 5-50, with Bahasa/Manglish AI, MyInvois e-invoicing, and DuitNow payments at a flat, transparent local-currency price (no HubSpot cliff, no Salesforce consultant tax).

The feature set — what to build, prioritized
P0 · Now (the wedge) 4
WhatsApp as the deal timeline (inbound + outbound via BSP) SEA channel Large build

Integrate one WhatsApp Business number per org through a BSP (e.g. local provider). Inbound messages auto-match contacts by phone and post as first-class activities on the deal timeline; reps send pre-approved templates and free-text replies inside the 24h customer-service window, all logged automatically. Reuses the existing polymorphic activities table (kind/externalId/deliveryStatus) and the EngineMailer webhook ingestion pattern.

Why it wins: This is THE channel in Malaysia (~84% reach, ~98% open) and the single biggest gap vs HubSpot/Salesforce, neither of which natively own WhatsApp. Regional WhatsApp-CRMs (Pepper Cloud, Rakan Sales) own the channel but lack the AI-native data layer and hide pricing. Owning the first-WhatsApp-message-to-follow-up gap is the highest daily-use pull in the segment.

AI auto-logging (fills-itself-in data layer) AI-native Medium build

Every inbound/outbound email (EngineMailer) and WhatsApp message auto-creates an activity and auto-links to the matching contact/deal — no manual entry. Auto-create contacts from unknown WhatsApp numbers / email senders, repurposing the existing extract-contact AI endpoint to parse fields from message content.

Why it wins: ~70% of CRM projects fail on adoption and 23% cite manual data entry as the #1 obstacle. The CRM that fills itself in (Attio's winning angle) is the most defensible, durable AI value — far more than chat/draft features, which are now table stakes. Directly converts in-flight email work into adoption fuel.

SEA-tuned AI drafting (BM / Manglish / Chinese / festive) AI-native Small build

Extend the existing draft-email endpoint with language+tone presets (Bahasa Malaysia, Manglish/code-switch, Mandarin, formal English) and festive-context awareness (Hari Raya, CNY, Deepavali, Christmas). Point the same engine at WhatsApp reply drafting, not just email.

Why it wins: 'Has AI' is commodity (Zia, Freddy, Breeze all draft English). The defensible angle is AI tuned to SEA selling — mainstream LLMs underperform in low-resource SEA languages and global CRMs treat them as bolt-ons. Low-effort prompt engineering on infrastructure that already exists; an instant feels-local demo moment.

CSV contact/company import with AI column mapping Onboarding Small build

Bulk import contacts/companies from CSV; repurpose the extract-contact AI endpoint to auto-map messy/foreign-language column headers to DealFlow fields.

Why it wins: No import path exists today — a brutal onboarding blocker. Simple CRMs go live in 1-2 weeks and ROI is gated on adoption; you can't adopt what you can't load. High-value, low-architecture-risk for a solo dev.

P1 · Next 7
AI multilingual thread summarization + per-contact relationship summary AI-native Small build

Summarize long multilingual WhatsApp/email threads and auto-generate a per-contact/deal 'here's everything that's happened with this account' summary from the auto-captured timeline.

Why it wins: Reuses the already-shipped summarize capability, grounded in DealFlow's own captured data (mirrors Attio/Salesforce summaries). High perceived value, near-zero new infrastructure, and a strong retention hook once the timeline auto-fills.

Next Action engine (follow-up nudges + silent-deal alerts) AI-native Medium build

Every deal gets a next-step + due date; a daily follow-up list surfaces what's due, and silent deals (no activity in N days) trigger a nudge. AI suggests the next best action from timeline context.

Why it wins: Follow-up is a memory problem — 80% of sales need 5+ touches but 44% quit after one. No incumbent at this tier turns the auto-captured timeline into a daily 'who to chase' list. Directly recovers leaked deals, the ROI story SMBs buy on.

MyInvois (LHDN) e-invoice export on Won deals SEA compliance Large build

When a deal is marked Won, generate an LHDN-compliant e-invoice payload (55 mandatory fields, TIN validation) and export/validate via MyInvois. Start with export + validation and a 'Won deal → compliant e-invoice draft' bridge; defer full real-time API and Peppol.

Why it wins: Legally forced adoption — Malaysia's MyInvois mandate now reaches the smallest firms (only sub-RM1m turnover exempt as of Jan 2026). Every target customer above RM1m MUST comply on a fixed deadline. Global CRMs ignore this entirely; it's a compliance wave DealFlow can ride as the GTM urgency hook.

Pipeline customization (stages CRUD + multiple pipelines) Core parity Small build

Add create/update/reorder/delete stage endpoints and a settings UI on top of the existing pipeline_stages table (orderIndex/winProbability/isWon/isLost already exist); allow a second pipeline. Stages are currently hardcoded and read-only.

Why it wins: A hard blocker today — SMBs in non-default verticals literally cannot adapt the pipeline. Small, self-contained build that removes a dismissal reason and is table-stakes parity with Pipedrive/Zoho.

PDPA-2024 trust layer (consent flag + audit log + residency) SEA compliance Small build

Add a per-contact consent field (captured? source? timestamp?), a basic audit log, breach-notification readiness, and explicitly market Supabase ap-southeast-2 data residency. Turn compliance into a trust feature.

Why it wins: PDPA 2024 (DPO, 72h breach notice, cross-border transfer rules) makes Malaysian buyers wary of US-hosted SaaS. DealFlow already runs in-region — low build effort, high trust signal that global tools and even local rivals under-surface.

Mobile-first responsive experience + SEA phone normalization Core parity Medium build

Ship a solid mobile web experience (reps live on phones) and normalize the free-text phone field with SEA country codes (critical for WhatsApp contact-matching).

Why it wins: Mobile is a quota multiplier (65% vs 22% quota attainment) and 'no mobile app' is Folk's #1 complaint; SEA selling is mobile/WhatsApp. Phone normalization is also a hard dependency for WhatsApp auto-matching. Cheap credibility that stops 'too lightweight' dismissal.

Transparent flat local-currency pricing + grant positioning Pricing/GTM Medium build

Single simple plan ~RM49/seat/mo (2 months free annual), published openly; thin free tier (1-2 seats, capped contacts/AI credits) as a funnel; 14-day no-card trial. Local payment rails (FPX, GrabPay, TnG, Boost) via an aggregator (HitPay/Tazapay). Market MSME Digital Grant MADANI 50%-off + HRD Corp claimable.

Why it wins: Owns the 'missing middle' — full-featured at the SMB entry band, explicitly avoiding HubSpot's 44x cliff + mandatory $1,500-$3,500 onboarding and Monday's seat-minimum games. Published pricing beats regional rivals who hide it; grants halve effective price and are themselves lead sources. Card-only Stripe leaks SEA conversions.

P2 · Later 3
DuitNow QR / FPX payment links on quotes & invoices SEA payments Medium build

Attach a DuitNow QR or DuitNow/FPX payment link to CRM-generated quotes/invoices so customers pay from the document; reconcile paid status back onto the deal.

Why it wins: Malaysia is #2 globally in QR-payment adoption (~61.5%); card/Stripe-first global CRMs miss the local rails entirely. Closes the deal-to-cash loop natively and pairs with the e-invoice feature for a complete won-deal-to-paid flow no incumbent offers locally.

Lightweight LLM Magic-Field enrichment (SEA-aware) AI-native Medium build

AI Attributes / Magic Fields that web-research and auto-fill company industry/size/description into a custom field. LLM-web-research only — no paid data-provider waterfall.

Why it wins: Folk/Attio-style enrichment that's cheap and good enough, and SEA-aware (recognizes local company suffixes like Sdn Bhd). Do NOT try to out-enrich Clay/Apollo (capital-heavy). Reinforces the fills-itself-in narrative at low build cost.

Natural-language search (wire up nlFilter) AI-native Small build

Wire the already-declared but unrouted nlFilter provider method into a /ai/filter route so users query records in plain language ('deals in negotiation closing this month, no activity in 7 days').

Why it wins: The infrastructure already exists and is wired to no route — nearly free to ship. A visible AI-native moment that competitors charge enterprise tiers for.

How a user actually uses it — workflows

A prospect messages the company WhatsApp asking for a quote; Mei Ling never has to open the CRM to log anything.
Mei Ling — sales rep at a 12-person KL office-furniture supplier
  1. Prospect sends a WhatsApp message to the company's business number: a mix of Bahasa and English asking about conference-table pricing.
  2. DealFlow ingests the message via the BSP webhook, matches the sender's phone to an existing contact (or auto-creates one), and posts it as an activity on the contact's timeline — zero manual entry.
  3. DealFlow auto-creates a deal in the 'Lead' stage and the AI extracts the requested product into the deal notes.
  4. Mei Ling opens the deal on her phone, taps 'Draft reply', picks the Manglish tone preset; the AI drafts a friendly quote follow-up referencing the conference table.
  5. She tweaks one line and sends; the outbound message is auto-logged and a Next Action ('send formal quote, due tomorrow') is set automatically.
  6. Next morning the daily follow-up list reminds her the quote is due; she sends it with a DuitNow QR payment link attached.

Friday pipeline review without chasing reps for updates.

  1. Hafiz opens the reports dashboard; because email and WhatsApp auto-log, every deal's last activity is current without reps updating anything.

  3. For each silent deal, he reads the AI per-deal relationship summary instead of scrolling the whole thread.
  4. He reassigns two stalled deals and the Next Action engine sets fresh follow-ups for the new owners.

Top strategic risks
  • Solo-builder bandwidth: the roadmap spans WhatsApp BSP integration, MyInvois compliance, payments, and AI — over-committing risks shipping nothing well. Mitigation = ruthless phasing (unblock email → adoption layer → WhatsApp → retention → compliance) and shipping one channel/number deeply before breadth.
  • WhatsApp/Meta platform dependency: BSP onboarding delays, template-approval rejections, per-message billing changes, or policy shifts can stall or raise the cost of the flagship wedge — outside the builder's control.
  • MyInvois compliance is a moving regulatory target: LHDN field requirements, deadlines, and API specs change; a wrong implementation carries customer compliance liability. Scope to export/validation first and track LHDN updates.
  • Regional incumbents (Pepper Cloud, Rakan Sales, Barantum, Second CRM) already own WhatsApp-first SEA and could add AI faster than a solo builder; global players (HubSpot/Zoho) could finally ship native WhatsApp + SEA languages.
  • Adoption risk persists even with auto-capture: ~70% of CRM projects fail and average adoption is ~26%; if onboarding isn't genuinely under-a-week and habit-forming, churn kills the ROI story regardless of features.
  • Support expectation as a differentiator cuts both ways: promising fast in-region human support is a moat only if the solo founder can actually sustain it as customers scale — it can become a time sink that starves the build.
  • Thin data undermines AI credibility early: new customers start with no history, so AI features must deliver value from message-1 (drafting, summarizing, auto-logging) rather than predictions, or the 'AI-native' promise rings hollow.
  • Pricing pressure: anchoring at ~RM49/seat against $9-14 global entry tiers (Bigin, Freshsales, Zoho) requires the AI+local value to be obvious; if the wedge isn't felt immediately, buyers default to cheaper commodity CRMs.

ai-crm

The AI-CRM race in 2026 has converged on three layers: (1) an embedded assistant/copilot that drafts emails and summarizes (commodity — HubSpot Breeze Assistant, Pipedrive Sales Assistant, Salesforce Sales Assistant all do this; DealFlow already has draft/summarize), (2) autonomous agents that run multi-step workflows (HubSpot 20+ Breeze Agents, Salesforce Agentforce SDR, Apollo agents — mostly enterprise-priced, $125-650/user/mo or per-conversation), and (3) the data layer: auto-enrichment + auto-capture that makes the CRM that fills itself in real. That third layer is where the genuine, dura…

  • The AI-CRM market has split into 3 layers — embedded assistant (commodity), autonomous agents (enterprise-priced, hype-heavy for SMBs), and the auto-fill data layer (auto-capture + auto-enrichment). The data layer is where real, defensible value lives and where Attio is winning with the CRM that fills itself in.
  • DealFlow existing AI (draft/summarize) is now table stakes — every competitor ships it. Necessary but not differentiating. Differentiation must come from the data layer (capture + enrichment) and SEA-tuning, not from more chat/draft features.
  • Predictive lead scoring and forecasting are largely hype for a NEW SMB CRM: they need 500+ contacts and 3+ months of clean history to beat manual scoring, suffer black-box trust issues, and amplify bad data. A solo builder should de-prioritize these until customers have data, and when built, make them glass box (show the reasoning).
  • Adoption, not AI sophistication, is the real battle. ~70% of CRM projects fail, 23% cite manual data entry as the top obstacle, average adoption is ~26%. The single highest-ROI AI feature is anything that removes data entry (auto-log emails/meetings/WhatsApp, auto-create contacts, auto-fill fields). This directly converts DealFlow pending email/EngineMailer work into adoption fuel.
  • The clearest SEA whitespace is the intersection no incumbent owns: auto-filling data layer + WhatsApp-first capture + SEA-tuned AI drafting (Bahasa Malaysia / Manglish / festive context). Global tools treat WhatsApp and SEA languages as bolt-ons; local WhatsApp-CRMs lack the AI-native data layer. PDPA consent handling is a required, trust-building constraint, not just compliance overhead.
  • Enrichment is a build-vs-buy decision a solo builder must respect: Clay/Apollo value is the waterfall across 75+ paid data providers (~15 credits/contact), which is capital- and integration-heavy. DealFlow should NOT try to out-enrich Clay; do lightweight LLM-web-research Magic-Field-style enrichment (like Folk/Attio AI Attributes) that is cheap, SEA-aware, and good enough, e.g. fill company industry/size/description from the web.
Opportunities:
  • Ship the fills-itself-in data layer as DealFlow flagship wedge: auto-log inbound/outbound email into activities (you already send via EngineMailer + track opens/clicks — extend to auto-create an activity per email and auto-link to the contact). This is the single highest-ROI, adoption-driving AI feature and directly reuses work in flight. Next, add lightweight LLM Magic Fields (Folk/Attio-style AI Attributes) that web-research and auto-fill company industry/size/description on a custom field — cheap, no paid data-provider waterfall needed.
  • Own WhatsApp-first capture for SEA: ingest WhatsApp Business messages into the same activity timeline and auto-create/match contacts by phone number. No global incumbent does this natively and it is THE channel in Malaysia. Pair with a built-in PDPA consent flag on contacts (consent captured? source? timestamp?) — turn compliance into a trust feature local rivals under-serve. This is a low-effort prompt-engineering win that mainstream LLMs and global CRMs do poorly, and a demoable feels-local moment that supports the SEA wedge.
  • De-prioritize (for now) predictive lead scoring, forecasting, and autonomous agents. They are the loudest marketing claims but deliver little for a new SMB CRM with thin data, carry black-box trust risk, and are expensive to build well. This sequence attacks the #1 reason CRMs fail (data-entry burden killing adoption) and lands the AI-native + local wedge simultaneously — both differentiators a solo builder can actually ship.
  • Treat AI summarize (already shipped) as a retention hook by auto-generating per-contact/deal relationship summaries from the auto-captured timeline (here is everything that has happened with this account), mirroring Attio/Salesforce Sales Assistant summaries but grounded in DealFlow own captured data — high perceived value, reuses existing summarize capability, no new data sources required.

dealflow-gaps

DealFlow is a competently built, multi-tenant B2B CRM with a clean Node/Fastify/Drizzle/Supabase + React/TanStack/Tailwind stack. The data model (15 tables in packages/db/src/schema) and API surface (apps/api/src/modules) cover the full CRM core: contacts, companies, deals with a drag-and-drop kanban pipeline, polymorphic activities (notes/tasks/emails), org-scoped custom fields on every entity, a fixed reports dashboard, team membership with owner/admin/member roles + token invitations, and a real email-send path through EngineMailer with open/click tracking via webhook. AI is present but sha…

  • CRM CORE IS SOLID AND COMPLETE for an SMB: contacts (packages/db/src/schema/contacts.ts), companies, deals with weighted-probability kanban stages + status auto-derivation on drag (apps/api/src/modules/deals/deals.repo.ts moveToStage), polymorphic activities enforcing one-parent via DB CHECK constraint (packages/db/src/schema/activities.ts), and a 5-KPI + 4-chart dashboard (apps/api/src/modules/reports/reports.repo.ts). This is a real working product, not a prototype.
  • MULTI-TENANCY AND SECURITY ARE GENUINELY STRONG, a real moat for SMB trust: every repo query is org-scoped (e.g. apps/api/src/modules/contacts/contacts.repo.ts), integration secrets are AES-256-GCM encrypted per-org (apps/api/src/modules/integrations/repo.ts), record-level ownership is enforced (apps/api/src/lib/authz.ts via assertCanWrite in every mutating route), webhook keys use constant-time compare (apps/api/src/modules/emails/engine-mailer-webhook.ts), and team invariants like last-owner use transactions + FOR UPDATE (apps/api/src/modules/members/repo.ts).
  • AI IS A BOLT-ON, NOT NATIVE -- this is the single biggest gap given the winning wedge is 'AI-native'. Only 3 endpoints exist (apps/api/src/modules/ai/routes.ts: summarize-activity, extract-contact, draft-email), each a manual button. There is no lead scoring, no next-best-action, no auto-activity-logging, no pipeline/deal-health insight, and no conversational/agentic layer. The provider interface even declares nlFilter (packages/ai/src/provider.ts) but it is wired to NO route -- natural-language search is unbuilt.
  • SEA-TUNING IS CURRENCY-ONLY -- the other half of the wedge is barely started. There is a full ISO-4217 currency catalog including MYR/SGD/IDR/THB/PHP/VND with a region->currency map and Accept-Language detection at signup (packages/shared/src/currency.ts, apps/api/src/modules/auth/service.ts). But outreach is EMAIL-ONLY (apps/api/src/modules/emails/routes.ts) with no WhatsApp -- the dominant SEA sales channel -- and there is no UI i18n, no Bahasa/Thai/Vietnamese, and no local tax/payment context.
  • PIPELINE IS NOT CUSTOMIZABLE: the 6 stages (Lead/Qualified/Proposal/Negotiation/Closed Won/Closed Lost) are hardcoded and seeded once at signup (apps/api/src/modules/pipelines/seed.ts); pipelines/routes.ts exposes only a GET. SMBs in different verticals cannot rename/reorder/add stages or create a second pipeline through the product.
  • DATA-IN AND DATA-OUT ARE THIN: no CSV/contact import, no enrichment, no inbound email capture or two-way mail sync (send-only), no webhooks/public API for customers, no automation/workflow engine, and reporting is a single fixed dashboard with no custom reports, forecasting, or per-rep leaderboards (apps/api/src/modules/reports/reports.repo.ts). Auth is password-only despite an oauth_accounts table existing (packages/db/src/schema/oauth-accounts.ts) -- no Google sign-in route is registered in apps/api/src/server.ts.
Opportunities:
  • MAKE AI NATIVE, NOT A BUTTON (highest-ROI wedge): the AI plumbing (per-org provider chain, fallback, prompts in packages/ai/src/providers/prompts.ts) is already built, so adding deal-health/lead scoring, next-best-action suggestions on contact/deal detail pages, and auto-summarized activity timelines is incremental work on existing infrastructure. Wiring the already-declared nlFilter (packages/ai/src/provider.ts) into a /ai/filter route would ship natural-language search cheaply.
  • ADD WHATSAPP AS A CHANNEL (highest-ROI SEA wedge): email is the only outreach channel (apps/api/src/modules/emails/routes.ts) but WhatsApp dominates SMB sales in Malaysia/SEA. The activities table is already polymorphic with kind/externalId/deliveryStatus columns (packages/db/src/schema/activities.ts) and there is a working webhook ingestion pattern (engine-mailer-webhook.ts) to copy for a WhatsApp provider -- so a WhatsApp send+log feature reuses most of the email architecture.
  • SHIP PIPELINE CUSTOMIZATION: stages are hardcoded in apps/api/src/modules/pipelines/seed.ts and read-only via the API. Adding create/update/reorder/delete stage endpoints + a settings UI is small, self-contained, and removes a hard blocker for SMBs in non-default verticals. The pipeline_stages table already has the needed columns (orderIndex, winProbability, isWon, isLost).
  • ADD CSV CONTACT/COMPANY IMPORT: there is no bulk import path; for a solo builder this is a high-value, low-architecture-risk feature that directly addresses CRM onboarding friction. The extract-contact AI endpoint (apps/api/src/modules/ai/routes.ts) could be repurposed to map messy columns.
  • UNBLOCK EMAIL END-TO-END before adding features: the known blocker (verify EngineMailer sending domain + paste the tracking webhook URL) gates the entire email + tracking value prop already built (resolveEmail in apps/api/src/modules/emails/routes.ts, webhook in engine-mailer-webhook.ts). Getting one real send + one tracked open verified should precede new build.
  • LOW-COST SEA POLISH: localized number/date/phone formatting and a Bahasa Malaysia UI pass would reinforce the 'local-tuned' positioning. Phone is currently a free-text field (packages/db/src/schema/contacts.ts) with no country/format handling -- adding SEA phone normalization is small and visible.
  • STABILIZE LOCAL DEV: the flaky local environment (noted in scope) is a tax on a solo builder's velocity; a documented one-command boot against Supabase would compound across every future feature.
